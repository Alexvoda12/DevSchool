Урок 1: Зачем нужно тестирование?

Почему важно тестировать код?

Тестирование — это процесс проверки того, что программа работает так, как задумано. Это не просто поиск ошибок, а гарантия качества.

Основные причины писать тесты:

  • Уверенность в коде: Тесты подтверждают, что код работает правильно.
  • Защита от регрессий: Изменения в одной части не ломают другую.
  • Документация: Тесты показывают, как использовать код.
  • Упрощение рефакторинга: Можно смело менять код, если тесты проходят.
  • Экономия времени: Лучше найти ошибку на этапе тестирования, чем в продакшне.

Виды тестирования

Тип тестов Что проверяют Пример
Юнит-тесты Отдельные функции/методы в изоляции Проверка функции сложения
Интеграционные тесты Взаимодействие между компонентами Проверка работы с базой данных
Функциональные тесты Работу системы в целом Тестирование всего веб-приложения
Нагрузочные тесты Производительность Сколько запросов выдержит сервер

Урок 2: unittest — встроенный модуль

Основы unittest

unittest — встроенный модуль Python для написания тестов, вдохновленный JUnit (Java).

Базовые принципы:

  • Тесты организуются в классы, наследующие от unittest.TestCase
  • Каждый тест — это метод, начинающийся со слова test_
  • Для проверок используются специальные assertion-методы
# Тестируемая функция
def add(a, b):
    return a + b

def divide(a, b):
    if b == 0:
        raise ValueError("Деление на ноль!")
    return a / b

# Файл с тестами: test_calculator.py
import unittest
from calculator import add, divide

class TestCalculator(unittest.TestCase):
    
    def test_add_positive(self):
        """Тест сложения положительных чисел"""
        self.assertEqual(add(2, 3), 5)
        self.assertEqual(add(10, 5), 15)
    
    def test_add_negative(self):
        """Тест сложения с отрицательными числами"""
        self.assertEqual(add(-1, 1), 0)
        self.assertEqual(add(-5, -3), -8)
    
    def test_add_zero(self):
        """Тест сложения с нулем"""
        self.assertEqual(add(0, 5), 5)
        self.assertEqual(add(3, 0), 3)
    
    def test_divide_normal(self):
        """Тест обычного деления"""
        self.assertEqual(divide(10, 2), 5)
        self.assertEqual(divide(7, 2), 3.5)
    
    def test_divide_by_zero(self):
        """Тест деления на ноль"""
        with self.assertRaises(ValueError):
            divide(10, 0)

if __name__ == '__main__':
    unittest.main()

Основные assertion-методы

Метод Проверка
assertEqual(a, b) a == b
assertNotEqual(a, b) a != b
assertTrue(x) bool(x) is True
assertFalse(x) bool(x) is False
assertIs(a, b) a is b

setUp и tearDown

Специальные методы для подготовки окружения перед каждым тестом и очистки после.

import unittest
import tempfile
import os

class TestFileOperations(unittest.TestCase):
    
    def setUp(self):
        """Выполняется перед КАЖДЫМ тестом"""
        self.temp_file = tempfile.NamedTemporaryFile(mode='w', delete=False)
        self.temp_file.write("Hello, World!")
        self.temp_file.close()
        self.filename = self.temp_file.name
    
    def tearDown(self):
        """Выполняется после КАЖДОГО теста"""
        os.unlink(self.filename)  # Удаляем временный файл
    
    def test_read_file(self):
        with open(self.filename, 'r') as f:
            content = f.read()
        self.assertEqual(content, "Hello, World!")
    
    def test_file_exists(self):
        self.assertTrue(os.path.exists(self.filename))

Также есть setUpClass и tearDownClass — выполняются один раз для всего класса (должны быть методами класса).

class TestDatabase(unittest.TestCase):

Урок 3: pytest — современный фреймворк

Почему pytest?

pytest — более современный и простой фреймворк для тестирования. Его преимущества:

  • Проще синтаксис: не нужно писать классы, просто функции
  • Мощные фикстуры: гибкая система подготовки данных
  • Параметризация: легко тестировать множество вариантов
  • Плагины: огромная экосистема
  • Подробные отчеты: красивые и информативные

Установка:

pip install pytest

Простые тесты с pytest

# Файл: test_calculator.py (для pytest)

def add(a, b):
    return a + b

def divide(a, b):
    if b == 0:
        raise ValueError("Деление на ноль!")
    return a / b

# Простые тесты-функции
def test_add_positive():
    assert add(2, 3) == 5
    assert add(10, 5) == 15

def test_add_negative():
    assert add(-1, 1) == 0
    assert add(-5, -3) == -8

def test_add_zero():
    assert add(0, 5) == 5
    assert add(3, 0) == 3

def test_divide_normal():
    assert divide(10, 2) == 5
    assert divide(7, 2) == 3.5

def test_divide_by_zero():
    import pytest
    with pytest.raises(ValueError, match="Деление на ноль!"):
        divide(10, 0)

Параметризация тестов

Позволяет запускать один тест с разными данными.

import pytest

def multiply(a, b):
    return a * b

@pytest.mark.parametrize("a,b,expected", [
    (2, 3, 6),
    (0, 5, 0),
    (-2, 4, -8),
    (10, 0, 0),
    (3, 3, 9),
    (1.5, 2, 3.0),
    (7, 8, 56),
])
def test_multiply(a, b, expected):
    assert multiply(a, b) == expected

# Тест будет запущен 7 раз с разными параметрами

Можно комбинировать несколько параметризаций:

@pytest.mark.parametrize("a", [1, 2, 3])
@pytest.mark.parametrize("b", [10, 20, 30])
def test_combination(a, b):
    # Запустится 3 * 3 = 9 раз
    assert a * b == a * b

# Именованные параметры для читаемости
@pytest.mark.parametrize("test_input,expected", [
    ("3+5", 8),
    ("2*4", 8),
    ("6//2", 3),
], ids=["addition", "multiplication", "division"])
def test_eval(test_input, expected):
    assert eval(test_input) == expected

Фикстуры (fixtures)

Фикстуры — это функции, которые подготавливают данные для тестов. Они заменяют setUp/tearDown из unittest.

import pytest
import tempfile
import os

@pytest.fixture
def temp_file():
    """Фикстура, создающая временный файл"""
    # Setup
    file = tempfile.NamedTemporaryFile(mode='w', delete=False)
    file.write("Test data")
    file.close()
    
    # Возвращаем имя файла в тест
    yield file.name
    
    # Teardown (выполняется после теста)
    os.unlink(file.name)

# Использование фикстуры
def test_read_file(temp_file):
    with open(temp_file, 'r') as f:
        content = f.read()
    assert content == "Test data"

def test_file_exists(temp_file):
    assert os.path.exists(temp_file)

Область видимости фикстур:

@pytest.fixture(scope="function")  # По умолчанию - для каждого теста
def function_fixture():
    return "Для каждого теста"

@pytest.fixture(scope="class")     # Один раз на класс
def class_fixture():
    return "Для всех тестов класса"

@pytest.fixture(scope="module")    # Один раз на модуль
def module_fixture():
    return "Для всех тестов в файле"

@pytest.fixture(scope="session")   # Один раз на всю сессию тестирования
def session_fixture():
    return "Для всех тестов в проекте"

Фикстуры могут использовать другие фикстуры:

Урок 4: Тестирование исключений и моки

Тестирование исключений

Важно проверять, что код правильно выбрасывает исключения в ошибочных ситуациях.

В unittest:

import unittest

def validate_age(age):
    if age < 0:
        raise ValueError("Возраст не может быть отрицательным")
    if age > 150:
        raise ValueError("Слишком большой возраст")
    return True

class TestValidation(unittest.TestCase):
    
    def test_negative_age(self):
        with self.assertRaises(ValueError) as context:
            validate_age(-5)
        self.assertEqual(str(context.exception), "Возраст не может быть отрицательным")
    
    def test_too_old(self):
        with self.assertRaisesRegex(ValueError, "Слишком большой возраст"):
            validate_age(200)

В pytest:

import pytest

def validate_age(age):
    if age < 0:
        raise ValueError("Возраст не может быть отрицательным")
    if age > 150:
        raise ValueError("Слишком большой возраст")
    return True

def test_negative_age():
    with pytest.raises(ValueError, match="Возраст не может быть отрицательным"):
        validate_age(-5)

def test_too_old():
    with pytest.raises(ValueError) as excinfo:
        validate_age(200)
    assert "слишком большой" in str(excinfo.value).lower()

Моки (unittest.mock)

Моки позволяют подменять части кода во время тестирования — например, чтобы не обращаться к реальному API или базе данных.

from unittest.mock import Mock, patch
import requests

# Функция, которая обращается к внешнему API
def get_weather(city):
    response = requests.get(f"https://api.weather.com/{city}")
    if response.status_code == 200:
        return response.json()
    return None

# Тест с моком
def test_get_weather():
    # Создаем мок для requests.get
    mock_response = Mock()
    mock_response.status_code = 200
    mock_response.json.return_value = {"temp": 20, "condition": "sunny"}
    
    with patch('requests.get', return_value=mock_response):
        result = get_weather("moscow")
        assert result == {"temp": 20, "condition": "sunny"}

# Более сложный пример
from unittest.mock import MagicMock

class EmailService:
    def send_email(self, to, subject, body):
        # Реальная отправка email
        pass

class UserService:
    def __init__(self, email_service):
        self.email_service = email_service
    
    def register_user(self, email):
        # ... регистрация пользователя
        self.email_service.send_email(email, "Welcome", "Thank you for registering!")
        return True

def test_register_user():
    # Создаем мок для email сервиса
    mock_email = MagicMock()
    service = UserService(mock_email)
    
    result = service.register_user("test@example.com")
    
    assert result is True
    # Проверяем, что метод был вызван с правильными аргументами
    mock_email.send_email.assert_called_once_with(
        "test@example.com", 
        "Welcome", 
        "Thank you for registering!"
    )

Основные возможности mock:

from unittest.mock import Mock

mock = Mock()

# Настройка возвращаемых значений
mock.get_data.return_value = [1, 2, 3]
assert mock.get_data() == [1, 2, 3]

# Настройка побочных эффектов (исключения)
mock.save.side_effect = ValueError("Ошибка сохранения")
try:
    mock.save()
except ValueError:
    pass  # Ожидаемо

# Проверка вызовов
mock.process(1, 2, key="value")
mock.process.assert_called_once()
mock.process.assert_called_with(1, 2, key="value")
mock.process.assert_called_once_with(1, 2, key="value")

# Количество вызовов
assert mock.process.call_count == 1

# Сброс мока
mock.reset_mock()

Урок 5: TDD — Test-Driven Development

Цикл Red-Green-Refactor

TDD (разработка через тестирование) — это методология, где тесты пишутся до кода.

RED

Написать тест, который падает

GREEN

Написать минимальный код, чтобы тест прошел

REFACTOR

Улучшить код, не ломая тесты

Пример TDD на практике:

Шаг 1 (RED): Пишем тест для функции, которой еще нет

# test_fizzbuzz.py
import pytest

def test_fizzbuzz_returns_fizz_for_3():
    from fizzbuzz import fizzbuzz
    assert fizzbuzz(3) == "Fizz"

# Запускаем тест — он падает, потому что функции нет

Шаг 2 (GREEN): Пишем минимальный код, чтобы тест прошел

# fizzbuzz.py
def fizzbuzz(n):
    return "Fizz"  # Минимальная реализация

# Теперь тест проходит

Шаг 3 (REFACTOR): Добавляем новые тесты и улучшаем код

# Добавляем новые тесты
def test_fizzbuzz_returns_buzz_for_5():
    from fizzbuzz import fizzbuzz
    assert fizzbuzz(5) == "Buzz"

def test_fizzbuzz_returns_fizzbuzz_for_15():
    from fizzbuzz import fizzbuzz
    assert fizzbuzz(15) == "FizzBuzz"

def test_fizzbuzz_returns_number_otherwise():
    from fizzbuzz import fizzbuzz
    assert fizzbuzz(1) == "1"
    assert fizzbuzz(2) == "2"
    assert fizzbuzz(4) == "4"

# Улучшаем реализацию
def fizzbuzz(n):
    if n % 15 == 0:
        return "FizzBuzz"
    if n % 3 == 0:
        return "Fizz"
    if n % 5 == 0:
        return "Buzz"
    return str(n)

# Все тесты проходят!

Преимущества TDD:

  • Код всегда покрыт тестами
  • Лучший дизайн API (сначала думаем, как использовать)
  • Меньше страха перед изменениями
  • Документация в виде тестов
  • Отлов ошибок на ранних этапах
